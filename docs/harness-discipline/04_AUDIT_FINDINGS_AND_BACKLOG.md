## 1. Executive summary

Spark's governance problem is not a missing charter. It is a charter **enforced only by honesty**.

The [`RUNTIME_CHARTER.md`](../../../../modules/spark-harness-core/source/docs/RUNTIME_CHARTER.md) already forbids — by name — every failure mode the founder is worried about: a route-specific regex owning execution authority (`RUNTIME_CHARTER.md:170`), a legacy patch or fallback router that can bypass or fight the Governor (`RUNTIME_CHARTER.md:171`), chat-only moves carrying actions, memory overriding fresh intent, and self-evolution mutating its own verifier. The schema layer in `spark-harness-core` is genuinely well-built: `kernel.py` binds ledgers to authorizations, gates executed statuses behind an `allow` verdict, and fail-closes protected self-evolution components. The mandates are correct and the record formats are sound.

The gap is that **the mandates are checked by hand-curated lists and caller-supplied booleans, not by code.** As the repos grew, that gap let five distinct pathologies compound:

1. **Parallel shadow-authority planes that decide BEFORE the Governor.** In the Telegram surface, `routeFirewall.ts` and `naturalRouteDecision.ts` are ~647 regex/keyword predicates that BOTH select the route AND hold a hard veto/grant. `telegramActionAuthority.ts:79-102` ANDs the regex verdict in front of the harness-core authorization, so a regex miss vetoes a Governor allow and a Governor allow is meaningless without the regex. The Governor is a wrapper around a prior regex verdict, not the origin of authority.

2. **The higher-intelligence layer is permanently shadowed.** `routeArbiter.ts` (the LLM classifier) is type-locked to `'off' | 'shadow'` with **no `'enforce'` value in the type** — the migration step became the permanent state. The regex keeps the wheel; the intelligent check can only write a JSONL telemetry record it never reads back.

3. **Self-certifying compliance flags.** `kernel.py:316-322` hardcodes `execution_boundary.legacy_authority_demoted = True` on every Governor decision regardless of whether any legacy plane was actually demoted, and `bridge_authority.py:121-222` *fabricates* a `governor-decision-v1` from the bridge's own verdict, then "verifies" against the decision it just authored. The record asserts compliance the system never checked.

4. **Benchmark-overfit canned answers in live code.** `domain-chip-memory` has grown a parallel answering plane — ~250+ literal `in question_lower` branches mapping specific eval phrasings (`"alice"`, `"a sunset with a palm tree"`, `"Liberal"`) to hardcoded answers — and `providers.py:1828` runs that rescue cascade **ahead of** real retrieval. A regex owns the answer the model should infer.

5. **God-files past the redline.** `cli.py` is 17,936 lines and `index.ts` is 10,486 — both over 5× and 3× the 3,000-line hard redline that explicitly named them. The line-count rule has no CI gate, so every route-by-route hotfix lands inline. The `spark-cli` branch tip is literally `hotfix/r28-longpath-guard` — 28 rounds of per-environment patches with no exit from the treadmill.

## 4. The remediation backlog

Prioritized in four waves. WAVE 0 ships the *checkers* first — because a self-evolving harness reads its repo as the norm, and until the floor is enforced, every other wave can be silently undone by the next evolution pass (broken-windows contagion). WAVES 1-3 then collapse the shadow planes, real-fix the masking/canned plane, and build the CUA escalation + observability spine.

Effort key: **S** ≤ 2 days · **M** ≈ 1 week · **L** > 1 week / multi-module.

### WAVE 0 — Stop the bleeding: ship the missing CI checkers

The single highest-leverage wave. None of these fix an offender directly; they make the offender *un-shippable going forward* and make the next evolution pass mechanically unable to add a 52nd. Lesson: Anthropic — make must-happen rules deterministic Stop-hooks, not advisory instructions (`https://code.claude.com/docs/en/best-practices`); Hermes — name escape hatches loudly so a lint can count them (`https://docs.openclaw.ai/gateway/security`).

## 6. Conflict-check & rollout sequencing (verified 2026-06-24)

Before enforcing this ruleset across all repos, it was stress-tested against the **new harness core and new systems** — `RUNTIME_CHARTER.md`, `SPARK_GENESIS_KERNEL_SCHEMA_DESIGN.md`, today's `SPARK_CONTROL_PROOF_PLAN_2026-06-24.md`, the `SPARK_SELF_AWARENESS_INTELLIGENCE_PLAN`, the live-chat-streaming + voice-runtime designs, and the shipped `harness-core/schemas/*.json`.

**Result: zero design-intent conflicts.** The new systems are *independent realizations* of this ruleset — the self-awareness intelligence module stamps every output `observability_non_authoritative`/`proposal_only` and is architecturally barred from self-authorizing (RL-12/R-25/PD-6); streaming drafts are presentation-only with the single final message as the sole audited artifact (RL-20); shipped schemas enforce RL-19/RL-20/RL-17/R-27 *structurally*. A ruleset derived from the charter does not fight the charter. **The set is safe to enforce in ratchet mode.** Four concrete refinements came out of the check and are folded into the waves:

### 6.1 One real conflict — and it's a contract bug the ruleset caught (WAVE 1)
RL-14 cannot be satisfied today because the dishonesty is in the **contract**: `governor-decision-v1.schema.json:57,65` makes `execution_boundary.legacy_authority_demoted` a *required* `{ "const": true }`, so `kernel.py:321` hardcodes `True` because no other value validates. **Fix: amend the schema** (`const: true` → `type: boolean`, or drop from `required`) — a protected-component change under `RL-17` (human approval) — then derive the flag from the live inventory. Add this to **W1.4**.

### 6.2 Sequencing: inventory before detector (WAVE 0 → WAVE 1 ordering is load-bearing)
The `RL-16` legacy-plane inventory (W0.5) **must land and be reconciled-from-source BEFORE** the `RL-01`/`RL-02` authority-site detector (W1.1) is switched to enforce. And the detector must flag a regex site **only when its return value is consumed as a precondition of execution** — *not* when it merely produces a label/evidence. Otherwise it false-positives legitimate advisory classifiers like the self-awareness module's `_classify_mode_and_intent` (`conversation_frame.py:190-209`) and `_implementation_route` (`capability_proposal.py:105-119`), whose only consumers are the eval harness and the observability recorder (no execution path is keyed on them). Those sites must be registered in the inventory as `disposition=evidence_adapter` so the detector allowlists them. This is exactly the legitimate-`narrow`-vs-forbidden-`grant` line from `R-02`.

### 6.3 Vocabulary reconciliation is a prerequisite for the schema-validation checkers (WAVE 0)
The schema-validation checkers (`RL-05`, `RL-20`) match records mechanically, but the corpus currently ships **three verdict spellings and two risk-tier enums**: the charter `allow/deny/interrupt/degrade`; the action-gate `allowed/blocked/requires_confirmation` (`conversation_frame.py:126-147`); the self-awareness doctrine `act/ask/explain/refuse` (`route_confidence_doctrine.py:8`); and risk tiers `high/destructive/external/credential/secret/...` vs the charter's `none/read/low/medium/high/critical`. Unify these to ONE canonical enum (an application of `R-16`) **before** the schema-validation checkers, or they cannot match cross-module records. Add as **W0.9**.

### 6.4 The voice-degrade gap the ruleset closes (WAVE 2/3)
The voice design (`VOICE_RUNTIME_ARCHITECTURE.md`) returns "text plus optional `voice_media`"; a `/voice` or voice-message turn where `voice_media` is missing/failed currently returns text silently. Under `RL-08`/`R-18`, that path must emit a `degrade` with a reason code (the failure must stay inspectable even though the user gets a graceful text reply). Not a conflict — a gap the ruleset closes; fold into the Track-3 surface-repair work.

### 6.5 The machine-origin decision — DECIDED (lane OFF), and a gated Wave 4
`AutonomyPolicyV1` ships a `machine_origin_policy` plane_type but **no self-grant field**, and `headless_defaults` force read-only/explicit-autonomy. The decision is made in [`06_AUTONOMY_AND_MACHINE_ORIGIN.md`](06_AUTONOMY_AND_MACHINE_ORIGIN.md): the lane is **OFF**, RL-04 is absolute, the default allowlist is empty, and high/critical/irreversible are never eligible. A 2026-06-24 red team (15 exploit paths) established that the lane is currently **unbuilt paper** — the only present safety property is the empty allowlist; every claimed protection (freeze-at-load, path-identity protection, provenance, reversibility, audit) is unbuilt.

**WAVE 4 — machine-origin enablement (GATED; do not start before WAVE 3 + the §4 preconditions).** This block ships *nothing* until the rest of the backlog is green. Its definition of done is the 11-item precondition set in `06` §4: **P-1** path-identity protection (RL-16/RL-17), **P-2** derived risk tier (RL-15), **P-3** `approval.source` provenance in `authorization-decision-v1` (RL-04), **P-4** freeze-at-load (RL-06), **P-5** effect-based mediation (RL-18), **P-6** cumulative per-run blast-radius budget, **P-7** run-level reversibility, **P-8** audit-latency ceiling + backpressure, **P-9** structured self-revoking allowlist (R-15/R-20), **P-10** anti-widening invariant (R-14/PD-7), **P-11** required-hook proof. Until then, machine-origin self-approval cannot occur.
